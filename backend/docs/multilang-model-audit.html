Variable Native-Language Support

This page visualizes the current backend source model and audits whether it can support users with different native languages while sharing the same target-language vocabulary data.

Audit verdict

Capable in product direction, but the current schema is only partially ready. Today, definitions and example translations are stored directly on language-specific rows, so Korean and Japanese learners can duplicate the same sense/example pool. The proposed normalized model fixes that by making target-language sense data shared and moving native-language text into translation tables.

1. Current Source Model

What the code stores today

target: words.language_code native: word_senses.definition_language_code native: examples.translation_language_code

The current source can store multiple languages, but the native language is part of the persisted sense/example rows instead of being a reusable side table.

Current implementation evidence

appendSenses inserts word_senses.definition_language_code. loadSenseOptions returns that same field. loadExamples reads examples.translation directly. The frontend DTO mirrors these fields exactly.

2. Current Data Shape

words

  • id
  • language_code = target language
  • normalized_text
  • part_of_speech
  • unique(language_code, normalized_text, part_of_speech)
->

word_senses

  • word_id
  • definition_language_code = native/display language
  • definition
  • short_definition
  • unique(word_id, definition_language_code, meaning_order)

examples

  • word_sense_id
  • sentence = target-language sentence
  • translation = native/display language
  • translation_language_code
  • difficulty_level
->

duplication effect

  • en charge + ko definition creates one sense row
  • en charge + ja definition may create another sense row
  • same target sentence may be regenerated per native language
  • LLM cost and quiz pool duplication grow with languages

3. Recommended Normalized Shape

words

  • language_code = target language
  • normalized_text
  • part_of_speech
->

word_senses

  • word_id
  • definition = canonical target-language definition
  • short_definition = compact canonical definition
  • unique(word_id, meaning_order)

sense_translations

  • word_sense_id
  • language_code = learner native/display language
  • definition
  • short_definition
  • unique(word_sense_id, language_code)
+

example_translations

  • example_id
  • language_code = learner native/display language
  • translation with optional **highlight** markers
  • unique(example_id, language_code)

examples

  • word_sense_id
  • sentence = target-language sentence
  • difficulty_level
  • source
  • cloze_text / cloze_answer
->

shared quiz pool

  • one target-language sentence set
  • many native-language translations
  • cheaper cache misses for future languages
  • stable word_sense_id for all learners

4. Lookup Flow After Normalization

1. Request Client sends target language and native/display language.
2. Load canonical Find shared words, word_senses, and target-language examples.
3. Join localization Left join sense_translations and example_translations for the requested native language.
4. Fill gaps If rows are missing and the enricher is configured, call translate outside the DB transaction and upsert cached translations.
5. Respond Return localized text when present, with canonical target-language fallback.

5. Capability Audit

Question Status Audit
Can the current source identify a target language? Yes words.language_code already distinguishes target-language vocabulary such as English, Korean, or Japanese words.
Can the current source identify a native/display language? Yes definition_language_code and translation_language_code exist today, and request DTOs include definition_language_code.
Can one sense be reused across many native languages? Partial Current uniqueness includes definition_language_code, so the same meaning can become multiple word_senses rows by native language.
Can one target sentence be reused across many native languages? Partial Current examples stores translation inline, so target sentences are not cleanly separated from localized translations.
Is the proposed normalized model capable? Yes It supports many native languages per target-language word sense without duplicating the learning unit or example sentence pool.
Should there be one global meaning object across all target languages? No Keep invariance scoped to one target-language word. Cross-target-language semantic identity is harder and not needed for this MVP.

6. Implementation Risks

API naming

Rename response semantics before implementation. definition_language_code currently sounds like a property of word_senses. In the normalized model it is the requested localization language.

Network calls

Do not call translation LLMs while holding a DB transaction. Load gaps, translate outside the transaction, then upsert in a short transaction.

Migration

Existing local rows can collide when dropping definition_language_code from the unique identity. Treat this migration as dev-reset-only unless a dedupe migration is written.

7. Recommended API Contract

Prefer explicit naming: language_code remains the target word language, localized_language_code is the requested native/display language, definition and short_definition are localized when available, and canonical fields should be omitted unless the frontend needs them.
