Project PN · Backend Design

Word Capture — Add-Word API & Data Model

A local-PostgreSQL vocabulary backend where words are captured (in-app passage selection or manual add), enriched once via a cheap hosted LLM, cached globally, and turned into personal learnable items.

Core principle: a word is global, but memory is personal. The learnable unit is a user_word_sense, never a raw word.
Go + chi PostgreSQL + pgcrypto pgxpool OpenAI-compatible LLM (env-configured) Single dev user (no auth yet)

Global / shared dictionary (enrichment cache) Personal / per-user memory

One row per meaning. Shared across all users.

  • cefr_level · meaning_order

Sense-specific example sentences (+ cloze).

  • id uuid
  • word_sense_id
  • sentence · translation
  • cloze_text · cloze_answer
personal

user_word_senses

The central learnable unit. One user ↔ one sense.

  • id uuid

  • id uuid

2Add-Word Flow (Lookup → Select → Add)

client

Capture or Manual Add

User selects a word in a passage or types one. Optional POS filter (default Any).

↓ POST /api/words/lookup
api

Normalize & cache lookup

Match words by full unique key when POS is concrete, else by (language_code, normalized_text) across all POS.

↓
cache hit

Return stored senses — free

No API call. Returns flattened sense_options[].

cache miss

LLM enrich once, then persist

Generate senses (scoped to POS if given) → insert words + word_senses + examples.

↓
client

SensePicker — choose exactly one

Add is disabled until one word_sense_id is selected. "Add best match" auto-picks the top-ranked sense (still a concrete id).

↓ optional: "None of these match"
force generate

Append a new sense

If an option was selected → send word_id + force=true (append under that exact word/POS). If none selected → a concrete POS is required. Never force=true with Any and no word_id.

↓ POST /api/learning-items {word_sense_id}
api · tx

Create the learnable item

Upsert user_word_senses (unique user+sense) + ensure one review_states row, in a single transaction.

4POS, Cache & Selection Rules

✓

POS is optional. Omitted or "Any" returns sense options across every POS; a concrete POS scopes results. "Any" is a lookup filter, never a learnable state.

✓

One API call per unique word/POS, ever. Cache hits are free DB reads; the LLM runs only on a true miss, and the result is shared globally.

✕

No item without a sense. POST /api/learning-items must receive a concrete word_sense_id — never raw text or a bare word_id (rejected 400/422).

✕

Illegal force. force=true with POS=Any and no word_id is rejected — the target word row is ambiguous.

✓

Reviews join the concrete sense. Quizzes derive from the stored word_sense_id → word_senses → words → examples. "Any" can never appear in review data.

5Validation Checklist

▢

lookup with POS omitted/Any returns multiple POS; concrete POS returns only that POS.

▢

force=true + Any + no word_id → 400; with word_id appends under that word/POS; with concrete POS generates under that POS.

▢

/api/learning-items rejects bodies without a concrete word_sense_id; accepts a valid one (idempotent on repeat).

▢

No row in words / word_senses / user_word_senses ever has POS = Any.

▢

go build ./... and go test ./... pass; migration runs cleanly.
