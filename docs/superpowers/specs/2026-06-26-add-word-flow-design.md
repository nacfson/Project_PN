# Add-Word Flow Redesign

**Date:** 2026-06-26
**Scope:** Frontend add-word flow for Project PN
**Status:** Design approved

## Goal

Make adding a word to the learning set as fast and frictionless as possible by removing manual sense selection and relying on AI to pick the best sense.

## Decisions Summary

| Topic | Decision |
|-------|----------|
| Sense selection | AI auto-picks the first/primary sense; user does not choose |
| Example sentences | One target-language example sentence per sense, generated by the backend enricher |
| Display language | Display-language meaning is shown in the backend lookup response but not emphasized in the add UI |
| User editing | None; users select only from AI-generated content |
| Multi-select | Not supported; one sense per add action |
| Deck selection | User selects the target deck before searching; default deck is pre-selected |
| Force-generate / wrong sense | Not handled inside the add-word flow |
| Container | Bottom-sheet modal on mobile; centered modal on web/desktop/Tauri |
| Confirmation | None; item is added silently after lookup |
| Success feedback | Toast: "Added {word} to {deck}" |
| Error handling | No error UI in the add-word flow; failures close the modal silently |
| Backend changes | None required |

## User Flow

1. User taps "Add word" from the learning list or home.
2. A bottom-sheet modal opens.
3. A deck selector at the top shows the currently selected deck (default deck pre-selected).
4. User types a word in the search input and submits (Enter or "Add Word" button).
5. Frontend calls `POST /api/words/lookup`.
6. Frontend selects `SenseOptions[0]` from the response.
7. Frontend calls `POST /api/learning-items` with the selected `word_sense_id` and selected `deck_id`.
8. Modal closes.
9. A toast confirms: "Added {word} to {deck}".
10. The learning list refreshes in the background.

## UI Components

### Bottom-Sheet Modal

- Opens from the bottom of the screen on mobile.
- Dismissible by swiping down or tapping the scrim.
- Becomes a centered dialog on web, desktop, and Tauri.

### Deck Selector Row

- Positioned at the top of the modal.
- Shows a deck icon, the deck name, and a "TARGET DECK" label.
- Tapping opens a deck picker or navigates to deck management.
- Default deck for the active target language is pre-selected.

### Search Input

- Uses the shared `Input` component.
- Auto-focused when the modal opens.
- Shows the `loading` spinner while `POST /api/words/lookup` is in flight.
- Submit triggers the add flow.

### Primary Button

- Label: "Add Word".
- Disabled until the search input is non-empty.
- Pressing Enter also submits.

### Toast

- Green pill at the bottom of the screen.
- Text: "Added \"{word}\" to {deck}".
- Auto-dismisses after approximately 2 seconds.

## Backend Interactions

The existing backend endpoints support this flow without modification.

### 1. Lookup the Word

```
POST /api/words/lookup
Authorization: Bearer <token>
{
  "text": "apple",
  "language_code": "en",
  "display_language_code": "ko"
}
```

Response includes `SenseOptions`, ordered by `meaning_order`. The frontend uses `SenseOptions[0]`.

### 2. Add the Learning Item

```
POST /api/learning-items
Authorization: Bearer <token>
{
  "word_sense_id": "<first-sense-id>",
  "deck_id": "<selected-deck-id>"
}
```

The endpoint is idempotent; re-adding the same sense succeeds silently.

### 3. Load Decks

```
GET /api/decks
Authorization: Bearer <token>
```

Called when the modal opens so the deck selector can show available decks and apply the default.

## Error Handling

- No error UI is shown inside the add-word modal.
- On lookup failure or add failure, the modal closes silently.
- The user may retry by reopening the modal and searching again.
- Success is the only user-visible outcome.

## Testing

### Frontend Tests

- Auto-pick logic selects `SenseOptions[0]`.
- Modal opens with the default deck pre-selected.
- Submitting the search calls `POST /api/words/lookup`, then `POST /api/learning-items` with the first sense and selected deck.
- Successful add closes the modal and shows the toast.
- Failed add closes the modal without showing a toast.

### Backend Compatibility

- Verify `POST /api/words/lookup` returns `SenseOptions` ordered by `meaning_order`.
- Verify `POST /api/learning-items` accepts `deck_id` and remains idempotent.

### Accessibility

- Search input has a clear placeholder and label.
- Deck selector is reachable and has an accessible label.
- Toast is announced by screen readers.

## Out of Scope

- Manual sense selection
- Multi-sense batch add
- Force-generate / "add another sense" inside the add flow
- User-editable definitions, examples, or notes
- Inline error messages or retry UI
- Source-sentence capture from the user
- Display-language translation fallback UI

## Open Questions

None. All design decisions are approved.
